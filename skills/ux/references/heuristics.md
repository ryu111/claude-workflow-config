# Nielsen's 10 Usability Heuristics

Jakob Nielsen 於 1994 年提出的 10 條啟發式原則，至今仍是 UX 設計的黃金標準。

---

## 1. Visibility of System Status 系統狀態可見性

**原則**：系統應始終讓使用者了解正在發生什麼，透過適當的回饋在合理的時間內提供。

### 範例

```
❌ 不良設計
[送出] ← 點擊後無任何反應，使用者不知道是否成功

✅ 良好設計
[送出中...] → [✓ 已送出] ← 清楚顯示狀態變化
```

### 實作要點

| 情境 | 回饋方式 |
|------|----------|
| 表單送出 | Loading spinner + 成功/失敗訊息 |
| 檔案上傳 | 進度條 + 百分比 |
| 頁面載入 | Skeleton screen |
| 背景處理 | Toast 通知 |
| 步驟流程 | Stepper 指示器 |

### 回饋時機

```
< 100ms   → 無需回饋（感覺即時）
100-1000ms → Spinner / 按鈕狀態變化
> 1s      → 進度指示器
> 10s     → 預估時間 + 取消選項
```

## 2. Match Between System and Real World 系統與真實世界的匹配

**原則**：使用使用者熟悉的語言、詞彙和概念，而非系統導向的術語。遵循真實世界的慣例，讓資訊以自然且符合邏輯的順序呈現。

### 範例

```
❌ 系統術語
"Error 404: Resource not found"
"Invalid input: regex mismatch"

✅ 使用者語言
"找不到這個頁面，它可能已被移動或刪除"
"請輸入有效的電子郵件地址，例如：name@example.com"
```

### 圖示設計

| 功能 | 推薦圖示 | 原因 |
|------|----------|------|
| 儲存 | 💾 磁碟片 | 約定俗成的符號 |
| 設定 | ⚙️ 齒輪 | 機械設備聯想 |
| 刪除 | 🗑️ 垃圾桶 | 真實世界對應 |
| 購物車 | 🛒 推車 | 購物經驗 |
| 首頁 | 🏠 房屋 | 「回家」概念 |

## 3. User Control and Freedom 使用者控制與自由

**原則**：使用者經常會誤選功能，需要清楚標示的「緊急出口」來離開不想要的狀態，而不必經過冗長的流程。支援復原 (Undo) 和重做 (Redo)。

### 必備功能

```
✓ Undo / Redo（復原 / 重做）
✓ Cancel（取消）
✓ Back（返回）
✓ Close（關閉）
✓ Skip（跳過）
✓ 隨時儲存進度
```

### 範例

```
❌ 無法回頭
購物結帳流程中，無法返回修改地址

✅ 充分控制
每個步驟都可返回上一步
已填寫的資料自動保留
可隨時修改購物車
```

### Modal 設計

```
┌─────────────────────────────┐
│  確定要刪除這個檔案嗎？       │
│                             │
│  檔案名稱：report.pdf        │
│                             │
│  [取消]      [刪除]         │ ← 明確的退出選項
└─────────────────────────────┘
```

## 4. Consistency and Standards 一致性與標準

**原則**：使用者不應該懷疑不同的詞彙、情境或動作是否代表相同的意思。遵循平台和產業慣例。

### 一致性類型

| 類型 | 說明 | 範例 |
|------|------|------|
| 內部一致性 | 產品內部統一 | 所有按鈕樣式相同 |
| 外部一致性 | 符合產業慣例 | Logo 點擊回首頁 |
| 視覺一致性 | 設計語言統一 | 色彩、字體、間距 |
| 功能一致性 | 相同操作相同結果 | 「X」永遠是關閉 |

### 範例

```
❌ 不一致
頁面 A: "送出"
頁面 B: "提交"
頁面 C: "確認"

✅ 一致
所有頁面統一使用 "送出"
```

### 遵循平台慣例

| 平台 | 慣例 |
|------|------|
| Web | Logo → 首頁、漢堡選單、麵包屑導航 |
| iOS | 返回在左上、滑動返回、Tab Bar |
| Android | 返回按鈕、Material Design |

## 5. Error Prevention 錯誤預防

**原則**：比起好的錯誤訊息，更好的設計是從一開始就預防問題發生。消除容易出錯的情況，或檢查後在使用者提交前提供確認選項。

### 預防策略

```
1. 消除 - 完全移除出錯可能
2. 限制 - 限制可用選項
3. 建議 - 提供推薦選項
4. 確認 - 提交前再次確認
```

### 範例

```
❌ 容易出錯
輸入日期：[         ] ← 自由輸入，格式不明

✅ 預防錯誤
輸入日期：[日期選擇器] ← 只能選擇有效日期
          [📅 2025/01/10]

❌ 危險操作無確認
[永久刪除] ← 直接執行

✅ 確認機制
[永久刪除] → "輸入 'DELETE' 以確認" → [確認刪除]
```

### 表單預防

| 技術 | 應用 |
|------|------|
| 輸入遮罩 | 電話：(___) ___-____ |
| 自動格式化 | 信用卡：1234 5678 9012 3456 |
| 選項限制 | Dropdown 代替自由輸入 |
| 即時驗證 | 輸入時檢查，非送出後 |
| 預設值 | 合理的預設減少輸入 |

## 6. Recognition Rather Than Recall 識別而非回憶

**原則**：透過讓物件、動作和選項可見，來最小化使用者的記憶負擔。使用者不應該需要記住資訊。使用說明應該在適當時候可見或容易取得。

### 範例

```
❌ 需要回憶
輸入產品代碼：[        ]
（使用者需要記住代碼）

✅ 可識別
選擇產品：
┌────────────────────┐
│ 🖼️ iPhone 15 Pro   │
│ 🖼️ iPhone 15       │
│ 🖼️ iPhone 14       │
└────────────────────┘
（看圖即可識別）
```

### 實作方式

| 技術 | 說明 |
|------|------|
| 最近項目 | 顯示最近使用的檔案/功能 |
| 搜尋建議 | 輸入時顯示匹配選項 |
| 範例格式 | "例如：name@example.com" |
| 預覽縮圖 | 檔案以視覺預覽呈現 |
| 脈絡提示 | Tooltip、佔位文字 |

## 7. Flexibility and Efficiency of Use 彈性與效率

**原則**：快捷鍵（新手使用者看不到）可以加速專家使用者的操作，使系統同時滿足新手和專家。允許使用者自訂常用動作。

### 加速器類型

```
新手路徑：選單 → 編輯 → 複製
專家捷徑：Cmd/Ctrl + C
```

| 類型 | 範例 |
|------|------|
| 鍵盤快捷鍵 | Ctrl+S 儲存 |
| 手勢操作 | 滑動刪除 |
| 自動完成 | 搜尋建議 |
| 預設值 | 常用選項預選 |
| 批次操作 | 多選後一次處理 |
| 範本 | 預設模板快速開始 |

### 漸進式揭露

```
基本模式（新手）：
┌─────────────────────┐
│ [簡單搜尋........] │
└─────────────────────┘

進階模式（專家）：
┌─────────────────────┐
│ [搜尋........]      │
│ 日期：[  ] ~ [  ]   │
│ 類型：[▼ 全部]      │
│ 狀態：☑進行中 ☑完成  │
└─────────────────────┘
     [顯示進階選項 ▲]
```

## 8. Aesthetic and Minimalist Design 美學與極簡設計

**原則**：介面不應包含不相關或很少需要的資訊。每一個額外的資訊單位都會與相關資訊競爭，降低其相對可見性。

### 原則

```
✓ 每個元素都應有存在的理由
✓ 移除不必要的裝飾
✓ 優先顯示重要資訊
✓ 使用留白創造層次
```

### 範例

```
❌ 資訊過載
┌─────────────────────────────────┐
│ 歡迎回來！今天是 2025/1/10     │
│ 您有 3 則未讀通知              │
│ 系統已更新至 v2.3.1            │
│ 點擊這裡查看新功能...          │
│ 訂閱電子報獲取最新消息...       │
│ 今日推薦：...                  │
│ 廣告：...                      │
│ ─────────────────────────────  │
│ [您實際想做的事情藏在這裡]      │
└─────────────────────────────────┘

✅ 聚焦核心
┌─────────────────────────────────┐
│ 您好，小明                      │
│                                 │
│ [+ 新增專案]                    │
│                                 │
│ 最近專案                        │
│ ├ 專案 A                       │
│ └ 專案 B                       │
└─────────────────────────────────┘
```

### 檢查問題

- [ ] 這個元素是使用者需要的嗎？
- [ ] 這個資訊是現在需要的嗎？
- [ ] 可以用更少的元素達成嗎？
- [ ] 哪些可以移到次要層級？

## 9. Help Users Recognize, Diagnose, and Recover from Errors 幫助使用者識別、診斷和修復錯誤

**原則**：錯誤訊息應以平實的語言表達（不使用代碼），精確指出問題，並建設性地建議解決方案。

### 錯誤訊息結構

```
1. 發生了什麼（問題）
2. 為什麼發生（原因）
3. 如何修復（解決方案）
```

### 範例

```
❌ 不良錯誤訊息
"Error 500"
"操作失敗"
"無效輸入"

✅ 良好錯誤訊息
┌─────────────────────────────────┐
│ ⚠️ 無法完成付款                  │
│                                 │
│ 您的信用卡已過期。               │
│                                 │
│ 請更新您的付款資訊，或選擇       │
│ 其他付款方式。                   │
│                                 │
│ [更新卡片]  [其他方式]           │
└─────────────────────────────────┘
```

### 錯誤預防 vs 錯誤處理

| 階段 | 策略 |
|------|------|
| 輸入前 | 格式提示、範例 |
| 輸入中 | 即時驗證、建議 |
| 輸入後 | 清楚的錯誤訊息 |
| 修復中 | 保留已填資料 |

## 10. Help and Documentation 幫助與文件

**原則**：即使系統最好不需要文件就能使用，仍可能需要提供幫助和文件。這類資訊應該易於搜尋、聚焦於使用者的任務、列出具體步驟，且不要太長。

### 幫助類型

| 類型 | 用途 | 範例 |
|------|------|------|
| 脈絡幫助 | 即時引導 | Tooltip、佔位文字 |
| 新手引導 | 首次使用 | Onboarding tour |
| 常見問題 | 自助服務 | FAQ 頁面 |
| 文件 | 深度學習 | 使用手冊 |
| 支援 | 人工協助 | Chat、客服 |

### 良好幫助的特點

```
✓ 任務導向（「如何...」）
✓ 可搜尋
✓ 步驟明確
✓ 包含範例
✓ 持續更新
```

### 漸進式幫助

```
Level 1: Tooltip
         "這個欄位用於..."

Level 2: 行內說明
         [?] 點擊查看更多說明

Level 3: 幫助文件
         完整的步驟指南

Level 4: 人工支援
         聯繫客服
```

## 啟發式評估方法

### 執行步驟

```
1. 訓練   - 熟悉 10 條原則
2. 獨立   - 3-5 人各自評估
3. 記錄   - 記錄每個問題
4. 評分   - 嚴重性評級
5. 彙整   - 合併去重
6. 排序   - 按嚴重性排序
7. 報告   - 產出改善建議
```

### 嚴重性等級

| 等級 | 說明 | 處理 |
|------|------|------|
| 0 | 不是可用性問題 | 忽略 |
| 1 | 表面問題 | 低優先 |
| 2 | 次要問題 | 排程修復 |
| 3 | 重大問題 | 高優先修復 |
| 4 | 災難性問題 | 必須立即修復 |

### 問題記錄模板

```markdown
## 問題 #001

**違反原則**: #1 系統狀態可見性
**嚴重性**: 3 (重大)
**位置**: 結帳頁面 - 送出按鈕

**描述**:
點擊「送出訂單」後，按鈕無任何狀態變化，使用者不知道訂單是否正在處理。

**建議**:
1. 點擊後顯示 loading 狀態
2. 成功後顯示確認訊息
3. 失敗後顯示錯誤原因
```

## Checklist

### 可見性與回饋
- [ ] 每個動作都有即時回饋
- [ ] 載入狀態清楚可見
- [ ] 進度可追蹤

### 語言與一致性
- [ ] 使用使用者熟悉的詞彙
- [ ] 術語在整個產品中一致
- [ ] 遵循平台慣例

### 控制與錯誤處理
- [ ] 提供復原/取消選項
- [ ] 錯誤訊息說明原因和解法
- [ ] 危險操作有確認機制

### 效率與幫助
- [ ] 有快捷鍵/手勢加速
- [ ] 提供脈絡相關的幫助
- [ ] 介面簡潔聚焦
