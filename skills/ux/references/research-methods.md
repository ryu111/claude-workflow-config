# 使用者研究方法

系統化收集使用者需求、行為和回饋的方法論。

---

## 研究方法概覽

### 方法分類

```
            定性研究                    定量研究
           (為什麼/如何)               (多少/多常)
    ┌────────────────────┬────────────────────┐
生  │   使用者訪談         │   問卷調查          │
成  │   脈絡訪查           │   A/B 測試          │
式  │   焦點團體           │   分析數據          │
    ├────────────────────┼────────────────────┤
評  │   可用性測試         │   點擊熱圖          │
估  │   啟發式評估         │   轉換率分析        │
式  │   認知走查           │   任務完成率        │
    └────────────────────┴────────────────────┘
```

### 選擇方法

| 目標 | 適合方法 |
|------|----------|
| 了解需求 | 訪談、脈絡訪查 |
| 驗證假設 | 可用性測試、A/B 測試 |
| 優化資訊架構 | 卡片分類、樹狀測試 |
| 評估設計 | 啟發式評估、5 秒測試 |
| 追蹤行為 | 分析數據、熱圖 |

## 1. 可用性測試 Usability Testing

> **觀察真實使用者完成任務的過程，找出可用性問題。**

### 測試類型

| 類型 | 說明 | 適用 |
|------|------|------|
| 有主持 | 研究員在場引導 | 深度洞察 |
| 無主持 | 遠端自行完成 | 大量測試 |
| 探索性 | 開放式任務 | 早期發現 |
| 比較性 | A vs B 比較 | 方案選擇 |

### 5 人法則

```
┌─────────────────────────────────────────┐
│ 可用性問題發現率                         │
│                                         │
│ 100% ┤                    ████████████  │
│  85% ┤              ██████              │
│  75% ┤        ██████                    │
│  50% ┤   █████                          │
│      └─────┬─────┬─────┬─────┬─────────│
│            1     2     3     4     5人  │
└─────────────────────────────────────────┘

→ 5 位使用者可發現約 85% 的可用性問題
→ 多次小規模測試優於一次大規模測試
```

### 測試流程

```
1. 準備階段
   ├── 定義目標
   ├── 設計任務
   ├── 準備原型/產品
   └── 招募參與者

2. 執行階段
   ├── 開場說明
   ├── 背景問題
   ├── 任務執行（Think Aloud）
   └── 結束訪談

3. 分析階段
   ├── 整理觀察
   ├── 識別問題
   ├── 優先排序
   └── 產出報告
```

### 任務設計原則

```
✓ 基於真實場景
✓ 明確但不指導
✓ 可在時間內完成
✓ 涵蓋關鍵流程

❌ 不良任務：「點擊設定按鈕」
✅ 良好任務：「你想更改帳戶密碼，請試著完成」
```

### Think Aloud Protocol 放聲思考法

```
提示語：
「請在操作時說出你的想法，
 包括你在看什麼、在想什麼、
 感到困惑或疑惑的地方。」

記錄：
- 說了什麼
- 做了什麼
- 停頓/猶豫
- 表情/肢體語言
```

## 2. 啟發式評估 Heuristic Evaluation

> **專家根據可用性原則檢查介面問題。**

### 評估流程

```
1. 準備
   ├── 選擇啟發式原則（如 Nielsen's 10）
   ├── 準備評估表
   └── 召集 3-5 位評估者

2. 獨立評估
   ├── 每位評估者獨立檢查
   ├── 記錄問題和違反原則
   └── 評估嚴重性

3. 彙整
   ├── 合併所有發現
   ├── 去除重複
   └── 優先排序
```

### 嚴重性評級

| 等級 | 說明 | 處理 |
|------|------|------|
| 0 | 非可用性問題 | 忽略 |
| 1 | 表面問題 | 低優先 |
| 2 | 次要問題 | 排程修復 |
| 3 | 重大問題 | 高優先 |
| 4 | 災難性問題 | 立即修復 |

### 評估報告模板

```markdown
## 問題 #001

**違反原則**: 系統狀態可見性
**嚴重性**: 3
**位置**: 結帳頁面 > 送出按鈕

**描述**:
點擊送出後無載入狀態，使用者不知訂單是否處理中。

**截圖**: [附圖]

**建議修復**:
1. 顯示 loading 狀態
2. 成功後跳轉確認頁
```

## 3. 卡片分類 Card Sorting

> **了解使用者如何組織和命名資訊。**

### 類型

| 類型 | 說明 | 適用 |
|------|------|------|
| 開放式 | 使用者自訂分類 | 探索結構 |
| 封閉式 | 使用既定分類 | 驗證結構 |
| 混合式 | 可修改分類 | 彈性探索 |

### 執行步驟

```
1. 準備卡片
   └── 每張卡片寫一個內容項目

2. 執行分類
   ├── 開放式：請使用者分成有意義的組別並命名
   └── 封閉式：請使用者將卡片放入預設類別

3. 分析結果
   ├── 相似性矩陣
   ├── 群聚分析
   └── 命名分析
```

### 範例

```
卡片內容（電商網站）：
┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐
│手機 │ │耳機 │ │沙發 │ │床墊 │
└─────┘ └─────┘ └─────┘ └─────┘
┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐
│電視 │ │冰箱 │ │桌子 │ │椅子 │
└─────┘ └─────┘ └─────┘ └─────┘

使用者分類結果：
├── 3C 產品：手機、耳機、電視
├── 家電：冰箱
└── 家具：沙發、床墊、桌子、椅子
```

### 分析產出

```
相似性矩陣：
         手機  耳機  電視  冰箱
手機      -    90%  70%  20%
耳機     90%   -    60%  15%
電視     70%  60%   -    40%
冰箱     20%  15%  40%   -

→ 手機和耳機高度相關，應放同一類別
```

## 4. A/B 測試

> **比較兩個版本，用數據決定哪個更好。**

### 測試流程

```
1. 假設
   └── 「將按鈕顏色從灰色改成綠色，會提高點擊率」

2. 設計
   ├── A 版本：原始（灰色按鈕）
   └── B 版本：變化（綠色按鈕）

3. 執行
   ├── 隨機分配使用者
   ├── 收集數據
   └── 達到統計顯著性

4. 分析
   ├── 比較指標
   └── 決定採用版本
```

### 關鍵指標

| 類型 | 指標 |
|------|------|
| 轉換 | 點擊率、註冊率、購買率 |
| 參與 | 停留時間、互動次數 |
| 留存 | 回訪率、流失率 |
| 滿意度 | NPS、CSAT |

### 注意事項

```
✓ 一次只測一個變數
✓ 樣本量足夠大
✓ 測試時間足夠長
✓ 考慮外部因素影響

❌ 常見錯誤：
- 過早停止測試
- 同時改變多個變數
- 忽略季節/時間因素
- 只看整體，不看分群
```

### 範例

```
測試：註冊按鈕文案

A 版本：「註冊」
B 版本：「免費開始」

結果：
┌─────────────────────────────────┐
│ 版本 A    │ 版本 B              │
│ 註冊率 3.2% │ 註冊率 4.8%        │
│ n=5000    │ n=5000              │
│ p < 0.05  │ ✓ 統計顯著          │
└─────────────────────────────────┘

結論：B 版本提升 50%，採用「免費開始」
```

## 5. 五秒測試 5-Second Test

> **測試使用者對頁面的第一印象和理解。**

### 執行方式

```
1. 展示頁面 5 秒
2. 隱藏頁面
3. 詢問問題：
   - 這個頁面是做什麼的？
   - 你記得看到什麼？
   - 最突出的元素是什麼？
   - 你會點擊什麼？
```

### 適用場景

| 測試目標 | 問題 |
|----------|------|
| 品牌印象 | 這是什麼類型的網站？ |
| 資訊層級 | 最重要的資訊是什麼？ |
| CTA 可見度 | 你會在哪裡點擊？ |
| 信任度 | 你信任這個網站嗎？ |

## 6. 眼動追蹤 Eye Tracking

> **了解使用者實際看向哪裡。**

### 產出類型

```
熱圖 (Heat Map)：
┌─────────────────────────────────┐
│ ████ Logo    [導航] [搜尋][👤] │
│ ██                             │
│ ████████████ 主標題            │
│ ████████                       │
│ ████ 內容...                   │
│ ██                             │
└─────────────────────────────────┘
↑ 紅色=注視最多，藍色=注視最少

凝視路徑 (Gaze Plot)：
1 → 2 → 3 → 4 → 5
Logo → 標題 → 圖片 → CTA → 內容
```

### F 型閱讀模式

```
┌─────────────────────────────────┐
│ ███████████████████████████████│  ← 水平掃描
│ ████████████████               │
│ ██████████                     │  ← 第二次水平
│ ███                            │
│ ██                             │  ↓ 垂直掃描
│ █                              │
│ █                              │
└─────────────────────────────────┘

→ 重要資訊放左上
→ 關鍵內容放在前兩段
```

## 7. 認知走查 Cognitive Walkthrough

> **模擬新使用者的思考過程，評估學習性。**

### 步驟

```
對於每個操作步驟，問：

1. 使用者會嘗試達成正確效果嗎？
   - 目標是否清楚？

2. 使用者會注意到正確的動作嗎？
   - 按鈕/連結是否可見？

3. 使用者會將動作與效果關聯嗎？
   - 標籤是否清楚？

4. 使用者會理解回饋嗎？
   - 是否知道成功了？
```

### 範例

```
任務：新增聯絡人

步驟 1：找到新增按鈕
├── Q1: 會想新增嗎？ ✓ 任務需要
├── Q2: 能看到按鈕嗎？ ✗ 按鈕太小
├── Q3: 知道這是新增嗎？ ✓ 有 + 圖示
└── Q4: 點擊後知道成功嗎？ ✓ 開啟表單

問題發現：新增按鈕可見性不足
建議：增大按鈕尺寸、增加對比
```

## 8. 問卷調查 Survey

### 常用量表

#### SUS (System Usability Scale)

```
10 道題目，5 點量表（1=非常不同意，5=非常同意）

1. 我想我會常常使用這個系統
2. 我覺得這個系統太複雜了 (反向)
3. 我覺得這個系統很容易使用
... (共 10 題)

計分：
- 奇數題：得分 - 1
- 偶數題：5 - 得分
- 總分 × 2.5 = SUS 分數 (0-100)

解讀：
> 80.3 = A (優秀)
> 68   = C (可接受)
< 68   = F (需改善)
```

#### NPS (Net Promoter Score)

```
問題：你有多大可能推薦這個產品給朋友？(0-10)

分類：
- 9-10：推薦者 (Promoters)
- 7-8：被動者 (Passives)
- 0-6：批評者 (Detractors)

NPS = 推薦者% - 批評者%

解讀：
> 50  = 優秀
> 0   = 良好
< 0   = 需改善
```

### 問卷設計原則

```
✓ 題目清楚簡潔
✓ 避免引導性問題
✓ 提供適當選項
✓ 控制題目數量
✓ 先簡單後複雜

❌ 避免：
- 雙重問題（一題問兩件事）
- 假設性問題
- 專業術語
- 太長的問卷
```

## 研究計畫模板

```markdown
# 研究計畫

## 背景
為什麼要做這個研究？

## 目標
要回答什麼問題？

## 方法
- 研究類型：可用性測試
- 參與者：5 人
- 時間：60 分鐘/人

## 參與者條件
- 年齡：25-45
- 使用經驗：有網購經驗
- 排除：公司員工

## 測試任務
1. 找到某商品並加入購物車
2. 完成結帳流程
3. 查看訂單狀態

## 時程
- 招募：Week 1
- 測試：Week 2
- 分析：Week 3
- 報告：Week 4

## 產出
- 問題清單（按嚴重性排序）
- 改善建議
- 影片重點剪輯
```

## Quick Reference

| 方法 | 何時用 | 參與者 | 時間 |
|------|--------|--------|------|
| 可用性測試 | 評估設計 | 5+ 人 | 1hr/人 |
| 啟發式評估 | 快速審查 | 3-5 專家 | 2-4hr |
| 卡片分類 | 資訊架構 | 15+ 人 | 30min/人 |
| A/B 測試 | 比較方案 | 1000+ 人 | 1-4 週 |
| 5 秒測試 | 第一印象 | 20+ 人 | 5min/人 |
| 問卷調查 | 大規模回饋 | 100+ 人 | 變動 |

## Checklist

### 測試前
- [ ] 定義明確的研究目標
- [ ] 選擇適當的研究方法
- [ ] 準備測試材料/原型
- [ ] 招募符合條件的參與者
- [ ] 準備訪談/測試腳本

### 測試中
- [ ] 確保參與者舒適
- [ ] 鼓勵放聲思考
- [ ] 保持中立不引導
- [ ] 記錄觀察和引述

### 測試後
- [ ] 整理原始資料
- [ ] 識別模式和洞察
- [ ] 按嚴重性優先排序
- [ ] 提出具體改善建議
- [ ] 分享研究發現
