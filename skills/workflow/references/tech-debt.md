# Technical Debt Cleanup

技術債清理原則。

## 核心原則

> **發現錯誤就要修正，不管是不是本次任務範圍！**

## 何時應該順手修復

| 發現問題 | 處理方式 |
|----------|----------|
| 明顯的 bug（不管是否在任務範圍） | ✅ 立即修復 |
| 不一致的程式碼模式 | ✅ 統一修復所有相關檔案 |
| 缺少的初始化或參數問題 | ✅ 修復並確保一致性 |
| 未使用的 import | ✅ 順手移除 |
| 過時的註解或文檔 | ✅ 更新或移除 |
| 重複的程式碼 | ⚠️ 記錄，視複雜度決定 |

## 執行原則

1. **發現即修復**：可以快速修復（< 5 分鐘）→ 立即處理
2. **一致性優先**：修復一個檔案 → 檢查相關檔案
3. **記錄複雜問題**：太複雜 → 記錄到 TODO 或 issues
4. **不破壞流程**：技術債清理是附加工作

## R→T 流程保障

技術債修復也要經過 R→T：

```
發現技術債
    ↓
修復（Main or Developer）
    ↓
🔍 REVIEWER 審查
    ↓
🧪 TESTER 測試
    ↓
✅ 完成
```

## 禁止的藉口

| 錯誤思維 | 正確思維 |
|----------|----------|
| ❌ 「這不是我寫的 bug」 | ✅ 「發現 bug = 我的責任修復」 |
| ❌ 「這是預存在的問題」 | ✅ 「所有問題都是專案的問題」 |
| ❌ 「不在本次任務範圍」 | ✅ 「現在修 5 分鐘，以後省 5 小時」 |
| ❌ 「這是別人的程式碼」 | ✅ 「專案的程式碼就是我的責任」 |

## 紅線

- ❌ 不要因為「不在任務範圍」而忽略明顯的 bug
- ❌ 不要留下「以後再修」的技術債
- ❌ 不要以「預存在問題」為由跳過修復
- ✅ 順手清理 = 長期維護成本降低
